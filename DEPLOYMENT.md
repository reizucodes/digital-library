# Deployment Guide

This guide covers deploying the Digital Library application to its three free-tier hosting targets: the Vue/Vite frontend to Vercel, the FastAPI backend to Render, and the PostgreSQL database to Neon.

---

## Overview

The Digital Library is a full-stack application split across three services. The PostgreSQL database is hosted on Neon and holds all persistent data. The FastAPI backend runs on Render, reads from Neon via a connection string, and exposes a REST API. The Vue frontend is a static build deployed to Vercel; it calls the Render backend URL at runtime via an environment variable baked in at build time.

---

## Prerequisites

Create free accounts on these three platforms before you start:

- **Neon** — https://neon.tech (PostgreSQL hosting)
- **Render** — https://render.com (backend hosting)
- **Vercel** — https://vercel.com (frontend hosting)

All three have free tiers that are sufficient for this project. You will also need your project repository accessible on GitHub, GitLab, or Bitbucket, as all three platforms deploy by connecting to a Git repo.

---

## Deployment Order

Deploy in this sequence: **database → backend → frontend**.

The reason is a chain of dependencies. The backend needs the Neon connection string before it can start. The frontend needs the Render backend URL before it can be built. Trying to deploy out of order means you will be setting placeholder values and redeploying, so do it in order the first time.

---

## 1. Neon Database Setup

1. Log in to https://neon.tech and click **New Project**.
2. Give the project a name (e.g., `digital-library`) and choose a region close to where your Render backend will run (US East is a safe default).
3. Neon creates a default database named `neondb` and a default role. Leave these as-is.
4. On the project dashboard, go to **Connection Details**. Select the **Connection string** tab and copy the full string. It looks like this:

   ```
   postgresql://alex:AbC123dEf@ep-cool-darkness-123456.us-east-2.aws.neon.tech/neondb?sslmode=require
   ```

   Keep this string somewhere safe. You will paste it into Render in the next step.

**Free-tier limits to know:**
- 0.5 GB storage, 1 compute unit, auto-suspends after 5 minutes of inactivity (cold start adds ~1–2 seconds to the first query after a period of idleness).
- One project per free account.

## 2. Render Backend Setup

### Connect your repository

1. Log in to Render and click **New > Web Service**.
2. Connect your GitHub/GitLab account and select the repository.
3. Render will detect `render.yaml` in `digital-library/backend/` and pre-fill the service settings. If it does not, set them manually:
   - **Root directory:** `digital-library/backend`
   - **Runtime:** Python
   - **Build command:** `pip install uv && uv sync --no-dev`
   - **Start command:** `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
   - **Health check path:** `/health`

### Set environment variables

In the Render dashboard for your service, go to **Environment** and add the following variables. `render.yaml` declares them with `sync: false`, which means Render expects you to fill in the values manually — they are intentionally not committed to the repo.

| Variable | Value |
|---|---|
| `DATABASE_URL` | Your Neon connection string (with `?sslmode=require`) |
| `CORS_ORIGINS` | Your Vercel frontend URL, e.g. `https://digital-library.vercel.app` |
| `SECRET_KEY` | Auto-generated by Render (listed in `render.yaml` with `generateValue: true`) |

For `CORS_ORIGINS`: if you do not yet have your Vercel URL, use a placeholder and update it after the frontend is deployed. The backend will start without it, but browser requests from the frontend will fail until it is correct.

For `SECRET_KEY`: Render generates this automatically on first deploy because of `generateValue: true` in `render.yaml`. You do not need to set it manually.

### First deploy

Click **Create Web Service**. Render will run the build command, then start the server. The first build takes 2–4 minutes. When the status shows **Live**, visit:

```
https://digital-library-api.onrender.com/health
```

You should get back `{"status": "ok"}` (or similar). If you get a 502, see the Troubleshooting section — Render free tier instances spin down after 15 minutes of inactivity.

### Run migrations

Migrations are not run automatically on deploy. After the first successful deploy, run Alembic manually from your local machine against the Neon database:

```bash
cd digital-library/backend
DATABASE_URL="postgresql://alex:AbC123dEf@ep-cool-darkness-123456.us-east-2.aws.neon.tech/neondb?sslmode=require" \
  uv run alembic upgrade head
```

Replace the connection string with your actual Neon URL. This creates the tables defined in `migrations/versions/`. You only need to do this once after first deploy, and again whenever you add a new migration file.

## 3. Vercel Frontend Setup

### Connect your repository

1. Log in to Vercel and click **Add New > Project**.
2. Import your repository from GitHub/GitLab.
3. Set the **Root Directory** to `digital-library/frontend`.
4. Vercel will detect the Vite framework and pre-fill these settings — confirm they are correct:
   - **Build command:** `npm run build`
   - **Output directory:** `dist`
   - **Install command:** `npm install`

### Set environment variables

In the project settings under **Environment Variables**, add:

| Variable | Value |
|---|---|
| `VITE_API_BASE_URL` | Your Render backend URL, e.g. `https://digital-library-api.onrender.com` |

This value is embedded into the JavaScript bundle at build time by Vite. If you change it later, you need to trigger a redeploy for the change to take effect.

### Deploy

Click **Deploy**. The build takes about 1 minute. When it finishes, Vercel gives you a URL like `https://digital-library.vercel.app`.

Go back to Render and update `CORS_ORIGINS` to this URL if you used a placeholder earlier. Render will automatically redeploy the backend when you save the environment variable change.

**SPA routing note:** `vercel.json` at the frontend root already handles Vue Router's history mode by rewriting all paths to `index.html`. You do not need to configure anything extra for deep-link URLs to work.

## Environment Variables Reference

### Backend (Render)

| Variable | Required | Example | Notes |
|---|---|---|---|
| `DATABASE_URL` | Yes | `postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/neondb?sslmode=require` | Must include `?sslmode=require` for Neon |
| `CORS_ORIGINS` | Yes | `https://digital-library.vercel.app` | Comma-separated for multiple origins, e.g. `https://digital-library.vercel.app,http://localhost:5173` |
| `SECRET_KEY` | Yes | *(auto-generated by Render)* | Used for token signing; Render generates this automatically |

### Frontend (Vercel)

| Variable | Required | Example | Notes |
|---|---|---|---|
| `VITE_API_BASE_URL` | Yes | `https://digital-library-api.onrender.com` | No trailing slash; embedded at build time |

## Running Migrations

Migrations are managed with Alembic. The migration files live in `digital-library/backend/migrations/versions/`.

**When to run migrations:**
- After the first production deploy (to create the initial schema).
- After adding a new migration file (schema changes).

**Command:**

```bash
cd digital-library/backend
DATABASE_URL="<your-neon-connection-string>" uv run alembic upgrade head
```

To check the current migration state:

```bash
DATABASE_URL="<your-neon-connection-string>" uv run alembic current
```

To generate a new migration after changing a SQLAlchemy model:

```bash
uv run alembic revision --autogenerate -m "describe the change"
```

Review the generated file in `migrations/versions/` before running it against production.

## Local Development

Local development uses SQLite and requires no changes to the deployment setup. The `.env.example` file in `digital-library/backend/` shows the local configuration:

```
DATABASE_URL=sqlite:///./data/library.db
CORS_ORIGINS=http://localhost:5173
SECRET_KEY=change-me-in-production
```

Copy it to `.env` and start the backend:

```bash
cd digital-library/backend
uv sync
uv run uvicorn app.main:app --reload
```

For the frontend:

```bash
cd digital-library/frontend
npm install
npm run dev
```

The frontend `.env.example` defaults to `VITE_API_BASE_URL=http://localhost:8000`, so local dev works out of the box without touching any production configuration.

SQLAlchemy handles both SQLite and PostgreSQL automatically based on the `DATABASE_URL` scheme — no code changes are needed when switching between environments.

## Troubleshooting

### CORS errors in the browser

**Symptom:** Browser console shows `Access to fetch at 'https://digital-library-api.onrender.com/...' from origin 'https://digital-library.vercel.app' has been blocked by CORS policy`.

**Cause:** `CORS_ORIGINS` on Render does not include the Vercel frontend URL, or it has a typo (trailing slash, wrong subdomain).

**Fix:** In the Render dashboard, go to Environment and verify `CORS_ORIGINS` exactly matches the Vercel URL with no trailing slash:
```
https://digital-library.vercel.app
```
Save and wait for the automatic redeploy.

---

### 502 Bad Gateway on Render

**Symptom:** Requests to the Render backend return a 502 error, especially after a period of no traffic.

**Cause:** Render free tier spins down instances after 15 minutes of inactivity. The first request after spin-down triggers a cold start that takes 30–60 seconds. During that window, Render returns 502.

**Fix:** This is expected behavior on the free tier. Wait 60 seconds and retry. If the 502 persists after a minute, check the Render logs (Dashboard > your service > Logs) for startup errors.

---

### Neon SSL errors

**Symptom:** Backend logs show `SSL connection has been closed unexpectedly` or `SSL SYSCALL error`.

**Cause:** The `DATABASE_URL` is missing `?sslmode=require`, or it is set to `sslmode=disable`.

**Fix:** Ensure the connection string ends with `?sslmode=require`:
```
postgresql://user:pass@ep-xxx.us-east-2.aws.neon.tech/neondb?sslmode=require
```
Update the `DATABASE_URL` environment variable on Render and redeploy.

---

### Vercel 404 on page refresh

**Symptom:** Navigating to `https://digital-library.vercel.app/books/42` directly (or refreshing the page) returns a 404.

**Cause:** Vue Router uses HTML5 history mode. Direct URL access hits Vercel's static file server, which looks for a file at that path and finds nothing.

**Status:** Already handled. `digital-library/frontend/vercel.json` contains a catch-all rewrite that sends all paths to `index.html`, letting Vue Router take over client-side. If you are seeing 404s, verify `vercel.json` is present at the frontend root and was not accidentally deleted.

---

### Migration errors on first deploy

**Symptom:** `alembic upgrade head` fails with `relation "books" already exists` or a similar conflict.

**Cause:** The migration was run more than once, or the table was created manually.

**Fix:** Check the current state first:
```bash
DATABASE_URL="<neon-url>" uv run alembic current
```
If it shows the latest revision, the schema is already up to date and no action is needed. If it shows no revision but the tables exist, stamp the current revision without running SQL:
```bash
DATABASE_URL="<neon-url>" uv run alembic stamp head
```
